# Mobile Content API

This document describes the recommended app-content surface for native clients.

Updated: 2026-03-18

The blog remains a static site generated by Deno + Lume. Native clients should
consume a static JSON API generated during the site build and served by CDN.

## Design Principles

- static output first, no dynamic backend required
- contract-first payloads, validated in CI
- one content source for web, iOS, Android, and HarmonyOS
- localized payloads using the same public language codes used in URLs
- additive evolution through explicit schema versioning
- native rendering first, without depending on HTML pages
- app-friendly offline caching from stable URLs

## Current Status In This Repository

The intended contract family already exists in `contracts/`:

- `contracts/app-manifest.schema.json`
- `contracts/posts-index.schema.json`
- `contracts/post-detail.schema.json`

However, the repository also still contains an older prototype:

- `contracts/post.schema.json`
- `plugins/content-contract.ts`
- `contracts/validate.ts`

Those older files describe a simpler post-only output and should be treated as
legacy scaffolding, not as the final API direction for the iOS app.

## Public Language Codes

Expose only these language codes to native clients:

- `en`
- `fr`
- `zh-hans`
- `zh-hant`

Do not expose internal TypeScript convenience keys such as `zhHans` or `zhHant`.

## Initial Endpoints

Recommended initial output:

```text
/api/app-manifest.json
/api/posts/index.json
/api/posts/<slug>.json
/fr/api/posts/index.json
/fr/api/posts/<slug>.json
/zh-hans/api/posts/index.json
/zh-hans/api/posts/<slug>.json
/zh-hant/api/posts/index.json
/zh-hant/api/posts/<slug>.json
```

## Why Feeds Are Not Enough

The existing RSS, Atom, and JSON Feed outputs remain useful for syndication, but
they should not be treated as the app API.

Reasons:

- feeds are currently limited in item count
- feeds optimize for subscribers, not app screens
- feeds do not provide a stable screen-oriented model
- feeds are awkward for alternates, offline state, and future pagination

## Contract Gaps To Close Before Freeze

### Rich Inline Content

The current `post-detail` study captures block structure, but text-bearing
blocks still flatten inline content to plain strings.

That is not sufficient for this blog because the Markdown already contains:

- inline links
- inline code
- emphasis / strong emphasis

Before the iOS app starts, extend the contract so paragraph-like content can
preserve inline semantics. Recommended direction:

- keep block-level structure for layout
- add inline children for paragraph, heading, list item, and quote text
- keep code blocks as dedicated block nodes

Example direction:

```json
{
  "type": "paragraph",
  "content": [
    { "type": "text", "text": "This site is built with " },
    { "type": "text", "text": "Lume", "marks": ["code"] },
    { "type": "text", "text": " and uses " },
    {
      "type": "link",
      "text": "a custom action",
      "url": "https://github.com/frenchvandal/aliyun-oss-cdn-sync-action"
    },
    { "type": "text", "text": "." }
  ]
}
```

This keeps the API native-friendly without forcing every client to consume raw
HTML.

### `updatedAt` Provenance

The current editorial model has a published date, but no separate shared
`updatedAt` field in post metadata.

Before freezing v1, make one explicit decision:

- `updatedAt` is optional until the editorial workflow supports it, or
- `updatedAt` becomes required and shared post metadata is extended to author it

Do not silently rely on `publishedAt === updatedAt` forever.

### Asset URL Policy

Native apps should not guess how to resolve relative asset paths. The contract
should define this clearly:

- `webUrl` stays absolute
- media URLs in `heroImage` and content blocks should be emitted as absolute
  HTTPS URLs, or
- the manifest should define one explicit origin clients must resolve against

The simpler and safer rule for mobile clients is to emit absolute asset URLs.

## Initial Contracts

### `app-manifest`

Purpose:

- bootstraps the app
- declares available languages
- points clients to the current index endpoints
- provides global schema and generation metadata

Suggested shape:

```json
{
  "version": "1",
  "generatedAt": "2026-03-17T10:00:00Z",
  "defaultLanguage": "en",
  "languages": ["en", "fr", "zh-hans", "zh-hant"],
  "postsIndex": [
    { "lang": "en", "apiUrl": "/api/posts/index.json" },
    { "lang": "fr", "apiUrl": "/fr/api/posts/index.json" },
    { "lang": "zh-hans", "apiUrl": "/zh-hans/api/posts/index.json" },
    { "lang": "zh-hant", "apiUrl": "/zh-hant/api/posts/index.json" }
  ]
}
```

Potential later additions if genuinely needed:

- `siteName`
- `supportUrl`
- `privacyUrl`
- `contentOrigin`

Do not add storefront metadata here unless the app runtime actually needs it.

### `posts-index`

Purpose:

- powers home and archive screens
- returns summaries only
- stays small and cacheable
- can seed local search later

Suggested shape:

```json
{
  "version": "1",
  "lang": "en",
  "items": [
    {
      "id": "alibaba-cloud-oss-cdn-deployment",
      "slug": "alibaba-cloud-oss-cdn-deployment",
      "title": "How This Blog Deploys to Alibaba Cloud OSS and CDN",
      "summary": "A practical walkthrough...",
      "publishedAt": "2026-03-10T00:00:00Z",
      "updatedAt": "2026-03-10T00:00:00Z",
      "readingTime": 6,
      "tags": ["devops", "github-actions", "alibaba-cloud"],
      "heroImage": null,
      "detailApiUrl": "/api/posts/alibaba-cloud-oss-cdn-deployment.json",
      "webUrl": "https://normco.re/posts/alibaba-cloud-oss-cdn-deployment/"
    }
  ]
}
```

Recommendation:

- keep `updatedAt` optional until editorial data supports it
- keep `heroImage` optional
- keep search server-side out of scope for v1

### `post-detail`

Purpose:

- powers the post detail screen
- includes localized metadata
- includes alternates
- includes native-renderable body content

Suggested shape:

```json
{
  "version": "1",
  "id": "alibaba-cloud-oss-cdn-deployment",
  "slug": "alibaba-cloud-oss-cdn-deployment",
  "lang": "en",
  "title": "How This Blog Deploys to Alibaba Cloud OSS and CDN",
  "summary": "A practical walkthrough...",
  "publishedAt": "2026-03-10T00:00:00Z",
  "updatedAt": "2026-03-10T00:00:00Z",
  "readingTime": 6,
  "tags": ["devops", "github-actions", "alibaba-cloud"],
  "alternates": [
    {
      "lang": "fr",
      "apiUrl": "/fr/api/posts/alibaba-cloud-oss-cdn-deployment.json",
      "webUrl": "https://normco.re/fr/posts/alibaba-cloud-oss-cdn-deployment/"
    }
  ],
  "heroImage": null,
  "webUrl": "https://normco.re/posts/alibaba-cloud-oss-cdn-deployment/",
  "blocks": [
    {
      "type": "paragraph",
      "content": [
        { "type": "text", "text": "This site is built with " },
        { "type": "text", "text": "Lume", "marks": ["code"] },
        { "type": "text", "text": "..." }
      ]
    },
    {
      "type": "heading",
      "level": 2,
      "content": [{ "type": "text", "text": "Pipeline at a glance" }]
    },
    {
      "type": "code",
      "language": "yaml",
      "content": "name: Deploy..."
    }
  ]
}
```

## Rendering Strategy

Use structured blocks as the primary rendering contract for native clients.

That lets each platform render content natively:

- SwiftUI on iOS
- Jetpack Compose on Android
- ArkUI on HarmonyOS

Recommended initial block families:

- paragraph
- heading
- code
- image
- quote
- list

Recommended initial inline families:

- text
- link
- emphasis mark
- strong mark
- inline code mark
- line break

If richer editorial features appear later, extend the contract explicitly
instead of leaking raw HTML requirements into every client.

## Search Strategy

Do not introduce a dedicated search endpoint for v1.

Recommended path:

- fetch localized `posts-index`
- cache it locally
- implement on-device title / summary / tag search in the app if needed

This keeps the first app fully static and avoids turning the blog into a search
backend before that complexity is justified.

## JSON Schema, Not OpenAPI First

The first implementation should use:

- JSON Schema for payload validation
- example JSON fixtures
- short Markdown endpoint documentation

OpenAPI can be added later if the API grows into:

- pagination and filtering
- richer error models
- query parameters
- remote search
- authenticated or personalized endpoints

For the current static CDN-served content model, JSON Schema is the simpler and
more useful contract layer.

## Cache Policy

Recommended defaults:

- `app-manifest.json`: short TTL
- index endpoints: medium TTL
- post detail endpoints: long TTL

Each payload should include:

- a schema `version`
- a generation timestamp
- a stable `id`

Optional later addition:

- a per-post `contentHash` if mobile cache invalidation becomes awkward with
  timestamps alone

## When To Extract `content-api`

Initially, JSON generation should stay inside the current web build.

Extract a dedicated `content-api` package only when:

- iOS is already consuming the contracts
- Android or HarmonyOS starts consuming them too
- the JSON generation rules become large enough to deserve their own module
- the web site and app API lifecycles start diverging materially

Until then, keeping generation in the web build is simpler and reduces
duplication.
