# Mobile Content API

This document describes the recommended content surface for native clients.

The blog remains a static site generated by Deno + Lume. Native clients should
consume a static JSON API generated during the site build and served by CDN.

## Design Principles

- static output first, no dynamic backend required
- contract-first payloads, validated in CI
- one content source for web, iOS, Android, and HarmonyOS
- localized payloads using the same public language codes used in URLs
- additive evolution through explicit schema versioning

## Public Language Codes

Expose only these language codes to native clients:

- `en`
- `fr`
- `zh-hans`
- `zh-hant`

Do not expose internal TypeScript convenience keys such as `zhHans` or `zhHant`.

## Initial Endpoints

Recommended initial output:

```text
/api/app-manifest.json
/api/posts/index.json
/api/posts/<slug>.json
/fr/api/posts/index.json
/fr/api/posts/<slug>.json
/zh-hans/api/posts/index.json
/zh-hans/api/posts/<slug>.json
/zh-hant/api/posts/index.json
/zh-hant/api/posts/<slug>.json
```

## Why Feeds Are Not Enough

The existing RSS, Atom, and JSON Feed outputs remain useful for syndication, but
they should not be treated as the app API.

Reasons:

- feeds are currently limited in item count
- feeds optimize for subscribers, not app screens
- feeds do not provide a stable screen-oriented model
- feeds are awkward for alternates, hero media, and future pagination

## Initial Contracts

### `app-manifest`

Purpose:

- bootstraps the app
- declares available languages
- points clients to the current index endpoints
- provides global schema and generation metadata

Suggested shape:

```json
{
  "version": "1",
  "generatedAt": "2026-03-17T10:00:00Z",
  "defaultLanguage": "en",
  "languages": ["en", "fr", "zh-hans", "zh-hant"],
  "postsIndex": [
    { "lang": "en", "apiUrl": "/api/posts/index.json" },
    { "lang": "fr", "apiUrl": "/fr/api/posts/index.json" },
    { "lang": "zh-hans", "apiUrl": "/zh-hans/api/posts/index.json" },
    { "lang": "zh-hant", "apiUrl": "/zh-hant/api/posts/index.json" }
  ]
}
```

### `posts-index`

Purpose:

- powers home and archive screens
- returns summaries only
- stays small and cacheable

Suggested shape:

```json
{
  "version": "1",
  "lang": "en",
  "items": [
    {
      "id": "alibaba-cloud-oss-cdn-deployment",
      "slug": "alibaba-cloud-oss-cdn-deployment",
      "title": "How This Blog Deploys to Alibaba Cloud OSS and CDN",
      "summary": "A practical walkthrough...",
      "publishedAt": "2026-03-10T00:00:00Z",
      "updatedAt": "2026-03-10T00:00:00Z",
      "readingTime": 6,
      "tags": ["devops", "github-actions", "alibaba-cloud"],
      "heroImage": null,
      "detailApiUrl": "/api/posts/alibaba-cloud-oss-cdn-deployment.json",
      "webUrl": "https://normco.re/posts/alibaba-cloud-oss-cdn-deployment/"
    }
  ]
}
```

### `post-detail`

Purpose:

- powers the post detail screen
- includes localized metadata
- includes alternates
- includes native-renderable body content

Suggested shape:

```json
{
  "version": "1",
  "id": "alibaba-cloud-oss-cdn-deployment",
  "slug": "alibaba-cloud-oss-cdn-deployment",
  "lang": "en",
  "title": "How This Blog Deploys to Alibaba Cloud OSS and CDN",
  "summary": "A practical walkthrough...",
  "publishedAt": "2026-03-10T00:00:00Z",
  "updatedAt": "2026-03-10T00:00:00Z",
  "readingTime": 6,
  "tags": ["devops", "github-actions", "alibaba-cloud"],
  "alternates": [
    {
      "lang": "fr",
      "apiUrl": "/fr/api/posts/alibaba-cloud-oss-cdn-deployment.json",
      "webUrl": "https://normco.re/fr/posts/alibaba-cloud-oss-cdn-deployment/"
    }
  ],
  "heroImage": null,
  "webUrl": "https://normco.re/posts/alibaba-cloud-oss-cdn-deployment/",
  "blocks": [
    { "type": "paragraph", "text": "This site is built with Lume..." },
    { "type": "heading", "level": 2, "text": "Pipeline at a glance" },
    { "type": "code", "language": "yaml", "content": "name: Deploy..." }
  ]
}
```

## Rendering Strategy

Use structured blocks as the primary rendering contract for native clients.

That lets each platform render content natively:

- SwiftUI on iOS
- Jetpack Compose on Android
- ArkUI on HarmonyOS

The initial block model can stay intentionally small:

- paragraph
- heading
- code
- image
- quote
- list

If richer editorial features appear later, extend the contract explicitly
instead of leaking raw HTML requirements into every client.

## JSON Schema, Not OpenAPI First

The first implementation should use:

- JSON Schema for payload validation
- example JSON fixtures
- short Markdown endpoint documentation

OpenAPI can be added later if the API grows into:

- pagination and filtering
- richer error models
- query parameters
- remote search
- authenticated or personalized endpoints

For the current static CDN-served content model, JSON Schema is the simpler and
more useful contract layer.

## Cache Policy

Recommended defaults:

- `app-manifest.json`: short TTL
- index endpoints: medium TTL
- post detail endpoints: long TTL with immutable fingerprinting later if needed

Each payload should include:

- a schema `version`
- generation timestamp
- stable `id`

That gives clients enough information to decide when migrations or refreshes are
needed.

## When To Extract `content-api`

Initially, JSON generation can stay inside `apps/web`.

Extract a dedicated `content-api` package only when:

- iOS is already consuming the contracts
- Android or HarmonyOS starts consuming them too
- the JSON generation rules become large enough to deserve their own module
- the web site and app API lifecycles start diverging materially

Until then, keeping generation in the web build is simpler and reduces
duplication.
