# Audit structurel du dépôt normco.re

**Date :** 2026-03-28
**Périmètre :** TypeScript, TSX, CSS, Lume, Deno, React/Ant Design, architecture
globale

---

## 1. Vue d'ensemble

### Architecture

Le dépôt est un site statique multilingue (en, fr, zh-Hans, zh-Hant) construit
avec :

| Couche | Technologie |
|---|---|
| Runtime / build | Deno 2.7.9 |
| Générateur statique | Lume 3.2.2 |
| Templating | TSX (Lume JSX) |
| Îlots interactifs | React 19 + Ant Design 6 |
| CSS | PostCSS + Lightning CSS, système de tokens `--ph-*` |
| Recherche | Pagefind |
| Déploiement | GitHub Actions → Alibaba Cloud OSS + CDN |
| App mobile | Android (Kotlin Compose, Material 3) |

### Structure des dossiers

```
_config/          # Modules de configuration Lume (plugins, feeds, processeurs…)
apps/android/     # Application Android Kotlin Compose
contracts/        # Schémas JSON + validation des contrats mobiles
docs/             # Documentation projet
plugins/          # Plugins Lume maison (shiki, content-contract, etc.)
scripts/          # Scripts utilitaires Deno (guards, lint, fingerprint…)
src/
  _components/    # Composants TSX partagés (Header, Footer, PostCard…)
  _includes/      # Layouts (base, post, tag)
  blog/client/    # Îlots React/Ant Design (App, ArchiveApp, PostApp, TagApp)
  posts/          # Contenus Markdown multilingues
  scripts/        # JS côté client (SW, prefetch, toggletips…)
  styles/         # CSS structuré en couches (@layer)
  utils/          # Utilitaires TS (i18n, feeds, types, slugify…)
test/             # Tests intégration
```

### Séparation des responsabilités

Le projet maintient une séparation claire :

- **Contenu** (Markdown + YAML frontmatter) séparé des **layouts** (TSX).
- **Tokens CSS** (`theme-tokens.css`) séparés des **composants CSS** (`layout.css`).
- **Données de vue blog** (`view-data.ts`) séparées des **composants React** (`App.tsx`).
- **Contrats mobiles** (`contracts/`) séparés de l'implémentation Android.
- **Configuration Lume** modulaire dans `_config/` (un fichier par responsabilité).

## 2. Points forts

### Typage exemplaire

- **Zéro usage de `any`** dans tout le code source.
- Usage systématique de `Readonly<T>` pour les props et structures de données.
- `as const` + types dérivés pour les constantes (`SUPPORTED_LANGUAGES`, etc.).
- Unions discriminées correctes pour les vues blog (`view: "archive" | "tag" | "post"`).
- Type guards dédiés dans `type-guards.ts`, utilisés de manière cohérente.
- `noUncheckedIndexedAccess` et `exactOptionalPropertyTypes` activés.

### Système de tokens CSS

- 93 custom properties `--ph-*` couvrant couleurs, typographie, espacement,
  layout, rayon, mouvement et focus.
- Espace colorimétrique OKLCH, `light-dark()` pour le thème, `color-mix()` pour
  les surfaces dérivées.
- Typographie fluide via `clamp()`, espacement responsive.
- CSS Layers (`@layer tokens, reset, base, layout, utilities`) empêchant les
  conflits de spécificité.
- Propriétés logiques (`inline-size`, `block-size`) pour la préparation RTL.

### Qualité de la configuration

- Git hooks via Lefthook (format + lint pre-commit, lint-commit).
- Design token guard (`scripts/design-token-guard.ts`) empêchant les valeurs
  CSS en dur.
- Typography guard pour la normalisation typographique.
- Validation de liens, d'imports browser-safe, et de contrats JSON.
- Budget de payload avec politique définie (`payload-policy.json`).
- CI/CD avec OIDC (pas de secrets longue durée).

### Couverture de tests

- 63 fichiers de test pour 109 fichiers source (~58 % de ratio
  fichiers testés).
- Tests co-localisés avec le code source (`Header_test.ts` à côté de
  `Header.tsx`).
- Usage de Faker avec seeds fixes pour la reproductibilité.

### Autres

- Aucun barrel export (`index.ts`) → pas de risque de dépendances circulaires.
- Imports explicites partout, tree-shaking friendly.
- Pas de styles inline dans le TSX (tout passe par des classes CSS).
- Service worker avec gestion offline et cache Pagefind.
- Documentation solide (ARCHITECTURE.md, README.md, AGENTS.md, docs/).

## 3. Problèmes majeurs

### P0 — Duplication de la logique d'archive

**Fichiers concernés :**

- `src/blog/client/App.tsx` (lignes ~340–410)
- `src/blog/client/ArchiveApp.tsx` (lignes ~47–110)
- `src/blog/archive-render.ts` (lignes ~19–90)

Trois implémentations parallèles des mêmes fonctions :

| Fonction | `App.tsx` | `ArchiveApp.tsx` | `archive-render.ts` |
|---|---|---|---|
| `formatArchiveMonth()` | ✓ (dayjs) | ✓ (Intl) | ✓ (Intl) |
| `groupArchiveMonths()` | ✓ | ✓ | ✓ |
| `ArchiveTimelineItem` | ✓ | ✓ | — (HTML string) |
| `let timelineIndex` | ✓ | ✓ | ✓ |

`App.tsx` utilise `dayjs` alors que les deux autres utilisent `Intl.DateTimeFormat`
natif — divergence d'implémentation qui peut engendrer des incohérences de formatage
entre routes.

Le compteur `let timelineIndex` muté dans un `.flatMap()` est un anti-pattern
fonctionnel présent dans les trois fichiers.

**Refactoring suggéré :** extraire `formatArchiveMonth`, `groupArchiveMonths` et
le type `ArchiveMonthGroup` dans un module partagé (par exemple
`src/blog/archive-common.ts`), en utilisant `Intl.DateTimeFormat` natif partout
pour éliminer la dépendance `dayjs` de `App.tsx`. Remplacer le compteur mutable
par un `.reduce()` ou un `.entries()` après aplatissement.

### P0 — Taille de `layout.css` (3 198 lignes)

Ce fichier concentre 200+ classes de composants (header, navigation, editorial,
blog-post, archive, feeds, popover, tooltip, notification, skeleton…). Il
représente un point de friction majeur pour la maintenabilité :

- Localiser un style spécifique demande de chercher dans ~3 200 lignes.
- Les conflits de merge sont fréquents quand plusieurs contributeurs touchent
  le styling.
- Pas de séparation entre composants indépendants.

**Refactoring suggéré :** découper en fichiers par famille de composants :

```
src/styles/
  components/
    header.css
    navigation.css
    editorial.css
    blog-post.css
    archive.css
    feeds.css
    popover.css
    blankslate.css
    ...
  layout.css  →  réduit aux grilles et shells globaux
```

Chaque fichier resterait dans `@layer layout` et serait importé depuis
`style.css`.

### P0 — Taille de `App.tsx` (1 111 lignes)

Ce fichier contient toutes les vues blog (Archive, Tag, Post) + composants
partagés + helpers. Il pourrait être découpé en :

- `ArchiveView.tsx` (~300 lignes)
- `TagView.tsx` (~150 lignes)
- `PostView.tsx` (~200 lignes)
- `BlogShell.tsx` (ConfigProvider, routing, breadcrumb)

Les composants partagés (`MetaLine`, `StoryTags`, etc.) sont déjà dans
`common.tsx` — bon réflexe. Le découpage des vues réduirait la charge cognitive
et faciliterait les reviews.

### P1 — Taille de `i18n.ts` (1 070 lignes)

Ce fichier mélange :

- Définitions de types et constantes de mapping linguistique (~200 lignes).
- Toutes les traductions des quatre langues (~700 lignes).
- Fonctions utilitaires de résolution de langue (~170 lignes).

**Refactoring suggéré :**

```
src/utils/
  i18n/
    types.ts           # SiteLanguage, mappings, exports
    translations.ts    # SITE_TRANSLATIONS (les ~700 lignes de données)
    helpers.ts         # getPageContext, getLocalizedUrl, etc.
    index.ts           # ré-export public (ou import direct de chaque fichier)
```

### P1 — Valeurs CSS en dur dans `layout.css`

Malgré l'excellent système de tokens, quelques fuites subsistent :

| Pattern | Occurrences | Token existant |
|---|---|---|
| `1px` (border-width) | ~30 | `--ph-border-hairline` |
| `letter-spacing: -0.025em` etc. | ~10 | Pas de token |
| `999px` (pill radius) | ~5 | Pas de token |
| `blur(16px) saturate(1.1)` | ~3 | Pas de token |
| `6px`, `12px` (gaps) | ~8 | `--ph-space-2`, `--ph-space-3` |

**Action :** ajouter les tokens manquants dans `theme-tokens.css` :

```css
--ph-letter-spacing-tight: -0.025em;
--ph-letter-spacing-snug: -0.015em;
--ph-letter-spacing-wide: 0.08em;
--ph-radius-pill: 999px;
--ph-backdrop-frosted: blur(16px) saturate(1.1);
```

Puis lancer `deno task design:guard` pour identifier les valeurs restantes.

### P1 — Tokens Ant Design générés non alignés

`src/styles/generated/antd-components.css` contient 109 couleurs hexadécimales en
dur (`#404040`, `#595959`, `#ff4d4f`…) qui ne sont pas connectées au système
`--ph-*`. Comme ce fichier est auto-généré par `scripts/generate-antd-css.ts`,
toute édition manuelle sera écrasée.

**Action :** configurer le script de génération Ant Design pour mapper les couleurs
de composants vers les tokens `--ph-*` existants, ou ajouter une passe de
post-traitement qui remplace les hex par les custom properties correspondantes.

## 4. Propositions de refactorings

### 4.1 Extraction de la logique d'archive (P0, effort faible)

**Avant (`App.tsx`, dupliqué dans 2 autres fichiers) :**

```tsx
function formatArchiveMonth(
  monthKey: string,
  locale: string,
): { label: string; shortLabel: string; year: number } {
  const parts = monthKey.split("-");
  // ... ~20 lignes avec dayjs dans App.tsx, Intl dans ArchiveApp.tsx
}

function groupArchiveMonths(
  posts: readonly BlogStoryCard[],
): ArchiveMonthGroup[] {
  // ... ~30 lignes dupliquées
}
```

**Après (`src/blog/archive-common.ts`, module partagé) :**

```tsx
export type ArchiveMonthGroup = Readonly<{
  key: string;
  year: number;
  label: string;
  shortLabel: string;
  anchorId: string;
  posts: readonly BlogStoryCard[];
}>;

export function formatArchiveMonth(
  monthKey: string,
  locale: string,
): Pick<ArchiveMonthGroup, "label" | "shortLabel" | "year"> {
  const [yearText = "0", monthText = "1"] = monthKey.split("-");
  const year = Number.parseInt(yearText, 10);
  const month = Number.parseInt(monthText, 10);
  const date = new Date(Date.UTC(year, month - 1, 1));
  return {
    year,
    label: new Intl.DateTimeFormat(locale, { year: "numeric", month: "long" })
      .format(date),
    shortLabel: new Intl.DateTimeFormat(locale, { month: "short" })
      .format(date),
  };
}

export function groupArchiveMonths(
  posts: readonly BlogStoryCard[],
  locale: string,
): readonly ArchiveMonthGroup[] {
  // implémentation unique
}
```

### 4.2 Remplacement du compteur mutable (P1, effort faible)

**Avant :**

```tsx
let timelineIndex = 0;
return months.flatMap((month) =>
  month.posts.map((story) => {
    timelineIndex += 1;
    return { ...item, index: timelineIndex };
  })
);
```

**Après :**

```tsx
const allPosts = months.flatMap((month) => month.posts);
return allPosts.map((story, i) => ({
  ...buildTimelineItem(story),
  index: i + 1,
}));
```

### 4.3 Découpage de `layout.css` (P0, effort moyen)

Créer un fichier par famille de composants, tous dans `@layer layout` :

```css
/* src/styles/components/header.css */
@layer layout {
  .site-header { /* ... */ }
  .site-header__wrapper { /* ... */ }
  /* ~250 lignes */
}
```

Importer depuis `style.css` :

```css
@layer tokens, reset, base, layout, utilities;

@import "./styles/antd/theme-tokens.css";
```

## 6. Métriques résumées

| Dimension | Score | Commentaire |
|---|---|---|
| Sûreté de typage | 10/10 | Zéro `any`, readonly systématique, strict activé |
| Architecture front | 8/10 | Bonne séparation, duplication d'archive à corriger |
| Système de tokens | 9/10 | Complet et moderne, quelques fuites de valeurs en dur |
| CSS | 7/10 | Excellentes fondations, fichier monolithique à découper |
| Tests | 7/10 | 58 % de ratio fichiers, bonne co-localisation |
| DX (experience dev) | 9/10 | Guards, hooks, lint, scripts clairs, bonne documentation |
| Configuration Lume/Deno | 9/10 | Modulaire, bien documentée, facile à étendre |
| Performance | 8/10 | Inline theme bootstrap, fingerprint, SW, peu d'état React |
| Maintenabilité | 8/10 | Solide, quelques gros fichiers à fragmenter |
| Documentation | 9/10 | ARCHITECTURE.md, README, AGENTS.md, docs/ complets |

**Verdict global :** c'est un dépôt de haute qualité, avec une discipline de
typage rare, un système de design tokens bien pensé et une configuration build
solide. Les axes d'amélioration sont principalement structurels (découpage de
fichiers trop volumineux) et hygiéniques (aligner les dernières valeurs CSS en
dur avec le système de tokens).
